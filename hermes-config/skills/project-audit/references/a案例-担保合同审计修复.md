# 案例：批量修复实体文件 Part2 内容单薄

> 背景：审计发现29个实体文件的Part2缺少实质性业务规则（WHERE筛选条件、DECODE/CASE WHEN映射）。
> 其中23个有对应源码（.prc文件），6个无源码或为汇总统计报表。

## 处理策略

### 策略1：合并同类项写入待优化清单

不要每条文件独立一条记录。将相同类型的问题合并为一条概括性记录：

```
// ❌ 坏做法：29条记录，每条一个文件
{"description": "Part2内容单薄: 个人贷款发生额信息_JS_201_GRDKFS.md", ...}

// ✅ 好做法：1条记录，涉及文件列包含全部29个
{"description": "Part2内容单薄（缺少实质性SQL筛选条件）：共29个实体文件",
 "files": "文件1、文件2、...文件29", ...}
```

### 策略2：分批处理—先处理有源码的

| 类别 | 数量 | 处理方式 |
|------|------|---------|
| 有对应 .prc 源码 | 20个 | 批量读取源码 → 提取WHERE/DECODE → 写入实体文件 |
| 五篇大文章汇总报表 | 6个 | 补充数据源说明（数据间接来自SMTMODS） |
| 无对应源码 | 3个 | 保持"未收录"标注，待确认 |

### 策略3：自动提取 vs 手动精修

批量提取源码关键信息，每个文件提取：
- 程序用途和业务域（头部注释）
- FROM/JOIN中的SMTMODS表列表
- WHERE条件（过滤掉元数据查询和JOIN条件）
- DECODE/CASE WHEN映射规则

提取后写入实体文件的Part2格式：

```
## 4. 业务筛选条件

**程序用途**: (从头部注释提取)
**SMTMODS 数据源表**: (从FROM/JOIN提取)
**时间筛选**: WHERE T.DATA_DATE = IS_DATE
**业务筛选条件**: (从WHERE子句提取)

## 5. 特殊处理规则

| 字段 | 规则 | 说明 |
|------|------|------|
```

### 注意事项

1. **WHERE 条件过滤**：自动提取时可能混入元数据查询和JOIN条件，需要在写入前人工审查
2. **DECODE/CASE WHEN 截断**：过长的规则应截断，避免markdown表格渲染异常
3. **汇总报表特殊处理**：五篇大文章类文件无独立.prc，不能按标准"内容单薄"判定

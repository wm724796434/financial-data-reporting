# 数据源合规：溯源分析方法

> 用于处理审计发现中的"缺集市表"和"数据源违规"类问题。
> 目标：确认程序的数据源是否最终来自 SMTMODS，并在实体文件中准确记录。

## 分析流程

### 步骤1：定位源码文件

从实体文件的 Part2.1（数据生成程序）获取存储过程名，在以下路径查找：

- `源码解析/加工层存储/<程序名>.prc` — 首选，含注释说明
- `源码/加工层存储/<程序名>.prc` — 原始备份（GBK 编码，需转换）

### 步骤2：提取所有 FROM/JOIN 表

读源码 30~200 行区间，提取所有 FROM 和 JOIN 子句中的表名：

| 表名 | Schema | 分类 |
|------|--------|------|
| `SMTMODS.L_ACCT_LOAN` | SMTMODS | ✅ 合规 |
| `L_CUST_C_TMP` | 默认/临时 | ⚠️ 间接引用 |
| `PBOCD_DATACORE.PBOCD_JS_205_CLZTX` | PBOCD_DATACORE | ❌ 非SMTMODS |
| `JS_102_TYKHXX_TEMP1` | 默认/临时 | ⏺️ 中间表 |

### 步骤3：分类判定

| 分类条件 | 标记 | 处理方式 |
|---------|------|---------|
| 主表直接 JOIN `SMTMODS.表名` | ✅ 合规 | 直接列出即可 |
| 主表来自 `L_*_TMP`、`TMP*` 等中间表，但中间表数据填充自 SMTMODS | ⚠️ 间接引用 | 在实体文件标注"数据最终来自 SMTMODS，但经过中间表，建议改直取" |
| 主表来自 `PBOCD_*`、`JS_*` 等业务接口表，且无 SMTMODS 上游 | ❌ 违规 | 必须整改，标注"未引用 SMTMODS，需确认上游链路" |
| 主表来自 config/映射表（名含 `CODE`、`CFG`、`MAP` 等） | 📋 配置表 | 可接受，标注说明 |
| 中间临时表（名含 `TMP`、`TEMP`、`TEMP1` 等） | ⏺️ 中间表 | 不单独标注，只说明其组成来源 |

### 步骤4：更新实体文件

在实体文件的"监管集市表来源"章节，替换原来的一句话描述为表格格式：

```markdown
## 2. 监管集市表来源

| 表名 | 角色 | 数据来源说明 |
|------|------|------------|
| `SMTMODS.L_ACCT_LOAN` | **主表**（贷款借据） | 筛选存量贷款借据 |
| `L_CUST_C_TMP` | 客户信息 | 从客户号获取详情（数据最终来自 SMTMODS.L_CUST_C）⚠️ 间接引用 |
| `PBOCD_DATACORE.PBOCD_JS_205_CLZTX` | 主表（存量再贴现） | ❌ 未经过 SMTMODS |

> ⚠️ **数据源合规**：本程序数据源来自 `PBOCD_DATACORE` schema 的接口表，**未直接引用 SMTMODS 监管集市表**。
```

### 特殊案例：汇总统计报表

对于五篇大文章类的汇总统计报表（无独立接口表、无独立 .prc）：

- 标注"本表为**汇总统计报表**（无对应接口表）"
- 说明其数据来自上游哪个接口表
- 说明上游接口表最终是否来自 SMTMODS
- 这类文件在审计中应排除"两缺"判定（因为不需要独立 .prc 和 SMTMODS 引用）

### 常见表名 Schema 对照

| 表名前缀 | Schema | 说明 |
|---------|--------|------|
| `SMTMODS.` | SMTMODS | ✅ 监管集市，合规 |
| `PBOCD_DATACORE.` | PBOCD_DATACORE | ❌ 业务接口数据 |
| `PBOCD_` | 默认 | ❌ 业务接口表 |
| `JS_` | 默认 | ❌ 业务接口表 |
| `L_CUST_C_TMP` | 默认 | ⚠️ 中间表（来自 SMTMODS.L_CUST_C 但经过中间层） |
| `L_*_TMP` | 默认 | ⚠️ 客户/账户信息中间层 |
| `*_CODE` | 默认 | 📋 映射/码值配置表 |
| `*_TMP*` | 默认 | ⏺️ 临时中间表 |

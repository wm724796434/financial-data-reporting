# 字段映射行内注释规范 — 存储过程/SQL 源码增强

## 来源

本规范来源于 `bsp_sp_js_201_hdaszdkfs.prc` 的试点样本实践。

## 目标

在源码解析文件的 SELECT 表达式中添加行内注释，使阅读代码时能**立即知道每个字段去了哪个目标字段、从哪里取数**，无需去数 INSERT INTO 的列顺序，也无需切出去查监管集市表定义。

## 注释格式

### 标准格式

```sql
表达式, -- → 目标字段  中文说明（来源/转换规则）
```

- `→ 目标字段` — 对应的接口表目标字段名（大写）
- `中文说明` — 该字段的业务含义
- `（来源/转换规则）` — 可选，说明数据来源或 CASE WHEN/DECODE 的映射逻辑

### 示例

**直接映射：**
```sql
T.CUST_ID, -- → CUST_ID  客户号（L_ACCT_LOAN.CUST_ID）
T.LOAN_ACCT_BAL, -- → LOAN_ACCT_BAL  贷款余额（L_ACCT_LOAN.LOAN_ACCT_BAL）
```

**计算字段：**
```sql
T.LOAN_ACCT_BAL * U.CCY_RATE, -- → LOAN_ACCT_BAL_RMB  贷款余额折人民币（余额×汇率）
```

**CASE WHEN 映射：**
```sql
CASE
  WHEN T.ACCT_TYP NOT LIKE '0301%' THEN 'C01' --单位贷款
  ELSE 'C03' --票据融资
END, -- → BIZ_TYPE  业务类别（非贴现=C01单位贷款，贴现=C03票据融资）
```

**固定值：**
```sql
'0', -- → DIGI_EFF_FLG  是否数字化效率提升业贷款（固定'0'，预留带*行业）
```

**SYS_GUID：**
```sql
SYS_GUID() AS REPORT_ID, -- → REPORT_ID  ID（系统生成GUID）
```

## 工作流程

### 第一步：定位输出目标

找到 INSERT INTO 语句，读取目标字段列表。这是 SELECT 表达式列序的依据。

```sql
INSERT INTO PBOCD_JS_201_HDASZDKFS
  (DATA_DATE,    -- ← 第1列
   FIELD_TYPE,   -- ← 第2列
   ORG_CODE,     -- ← 第3列
   ...           -- ← 第N列
  )
```

> ⚠️ 不要仅靠 INSERT 列名顺序来推 SELECT 列序，SELECT 表达式可能优化顺序或省略别名。必须逐位对照。

### 第二步：逐列匹配 SELECT 表达式

将 SELECT 列表中的每个表达式按位置对应到 INSERT 的目标列。

**关键原则**：SELECT 表达式的**位置顺序**决定了它存入哪个目标字段（列名不重要，INSERT INTO 是按位置而非名称匹配的）。

### 第三步：查监管集市表定义获取中文名

对于每个来源字段（如 `L_ACCT_LOAN.LOAN_ACCT_BAL`），到 `参考资料/监管集市/` 下查对应的表定义文档，获取字段中文名。

例如：
- `L_ACCT_LOAN` → `065_SMTMODS_L_ACCT_LOAN_贷款借据信息表.md`
- 字段 `LOAN_ACCT_BAL` → 中文名"贷款余额"

### 第四步：处理计算逻辑和 CASE WHEN

计算字段在注释中说清楚计算规则：

```sql
-- 金额非贴现取余额，贴现取放款金额减贴现利息
CASE
  WHEN A.ACCT_TYP NOT LIKE '0301%' THEN A.LOAN_ACCT_BAL
  ELSE A.DRAWDOWN_AMT - NVL(A.DISCOUNT_INTEREST, 0)
END
```

在注释中用简短一句话概括核心逻辑，而不是复制整个 CASE WHEN。

### 第五步：注意别名断层

这是一个关键准确性问题。代码中同一个字段在不同位置可能被赋予不同别名。

**示例**：临时表 `PBOCD_JS_201_HDASZDKFS_TMP1` 的 INSERT SELECT 中列名为 `DIGITAL_ECONOMY_INDUSTRY`，但在后续汇总查询中却被引用为 `T.FLAG`。

遇到此类别名断层：
1. 不要假设两处列名一致
2. 在注释中标注别名映射关系
3. 如果可能，核实临时表的建表 DDL 确认真实列名

### 第六步：多层嵌套的子查询

当外层 SELECT 和内层子查询都有字段需要注释时（如 T01/T02-T05 汇总），每层分别标注：

- 外层 SELECT：标注最终映射到接口表的哪个字段
- 内层子查询：标注计算字段的业务含义和来源

### 第七步：验证

修改完成后，重新读取修改部分，逐行核实：
1. 目标字段名是否与 INSERT INTO 列名一致
2. 中文说明是否准确（对照监管集市表定义）
3. 计算逻辑描述是否准确

## 需要参考的资料

1. **实体文件** — 获取输出接口表名和引用的监管集市表清单
2. **监管集市表定义** — 获取字段中文名、码表说明 (`参考资料/监管集市/xxx.md`)
3. **发文原文** — 获取接口表字段的官方填报说明 (`参考资料/发文原文/xxx.md`，辅助确认字段定义)

## 覆盖范围

每个源码解析文件中有几个 INSERT INTO 块，就有几个需要加注释的 SELECT 块：

| 类型 | 说明 |
|------|------|
| 临时表填充（INSERT INTO xxx_TMP） | 可选，只有被后续引用的字段需要注释关键别名 |
| 主输出汇总（FIELD_TYPE=T01/T02等） | 全部字段需要注释 |
| 主输出逐笔（FIELD_TYPE=T99等） | 全部字段需要注释，字段最多 |

## 试点样本参考

试点文件：`源码解析/加工层存储/bsp_sp_js_201_hdaszdkfs.prc`

覆盖了 3 个 INSERT INTO 块，添加了约 60 条字段映射注释。

## 准确第一原则

- **宁可少注，不能错注**：不确定的字段不强行注释
- **先查再注**：先在监管集市表定义中确认字段中文名，再写入注释
- **验证机制**：每次修改后重读变更部分，逐行核对
- **标注不确定性**：如果某个映射关系是推断的，标注"（推断，待确认）"

# 金数源码项目查询协议

> 本文件补充 `structured-repo-query` 技能，记录金数源码（金融数据源码）项目的特定三层查询协议。在涉及该项目的查询会话中加载本参考文件。

---

## 项目结构

```
金数源码/
├── 首页/index.md           ← 一级路由入口（49个报表实体索引）
├── 实体/                    ← 基于金融基础数据系统原文的报表实体文件
├── 源码解析/                ← 带增强注释的源码副本
│   ├── 加工层存储/          ← .prc 文件（主数据生成）
│   ├── 加工层特殊处理/      ← .sql 文件（数据修正/补录）
│   └── 应用层特殊处理/      ← .sql 文件（数据汇总/引用）
├── 参考资料/监管集市/        ← 191个监管集市表的字段级定义
├── 参考资料/金融基础数据系统/ ← 金融基础数据采集规范原文（49个）+ PDF补充（4个）
└── 金数源码/                ← 原始源码备份（禁止修改）
```

---

## 三层查询流程详解

### 第一层：首页/index.md（一级路由）

**作用**：定位目标报表/程序，获取实体文件路径。

**三种查询场景**：

| 场景 | 用户问法 | 操作 |
|------|---------|------|
| 场景A | "XX报表是干什么的" | 按业务域分组（贷款类、存款类、客户信息类、票据类等）找到目标报表 → 点实体文件链接 |
| 场景B | "XX程序的功能" | 查 index.md 末尾"按源码文件反向索引"章节 → 找到该程序属于哪些报表 → 点对应实体文件链接 |
| 场景C | "某个表字段的取数逻辑" | 走场景A/B定位报表实体 → 第二层获取概述 → 第三层查源码 |

**重要规则**：禁止在未查 index.md 之前直接用 glob/grep 搜索文件。index.md 已收录全部 67 个源码文件到 52 个报表实体的映射关系。

### 第二层：实体文件（功能/概述层）

**作用**：获取报表用途说明、数据流转、涉及的所有源码文件、监管集市表依赖关系。

**实体文件结构**：

```markdown
# 报表中文名（接口表名）
> 来源：[原文文件链接]

## 报表说明（精简总结，1-2句话）
## 业务分类（接口表代码、中文名、章节号）
## 关键逻辑（从源码提取的CASE WHEN映射 + 业务判断逻辑 + 关键变更）
## 特殊处理规则（从加工层特殊处理+应用层特殊处理SQL提取的原始注释）
## 涉及源码文件（按角色：数据生成/数据修正/数据引用）
## 引用的监管集市表（源码中引用的SMTMODS表）
## 字段清单（指引到原文文件，不复制字段表）
```

**限制**：实体文件只回答"报表的内容及概述"，不包含字段级取数逻辑。

### 第三层：源码解析文件（字段级取数逻辑）

**触发条件**：用户追问某个表字段的具体取数逻辑时。

**查找步骤**：
1. 读取 `源码解析/<子目录>/<文件名>`（.prc 或 .sql）
2. 从 SQL 代码中查找对应字段的映射逻辑（`SELECT ... LOAN_DUE_DATE ...` 或 `CASE WHEN` 块）
3. 结合 `参考资料/监管集市/` 下的表定义文档获取字段中文名和码表说明

## 字段级取数逻辑解释模板

当回答"XX字段的取数逻辑是什么"时，按以下模板组织回答：

### 1. 三层路由路径

```
第一层 → 首页/index.md → XX报表
第二层 → 实体/XX.md → 主数据生成源：bsp_sp_js_xxx.prc
第三层 → 源码解析/加工层存储/bsp_sp_js_xxx.prc
```

### 2. 接口字段定义

展示接口表字段名、中文名、类型、原文说明的表格。

### 3. 当前取数逻辑（SQL代码）

展示当前生效的 CASE WHEN 代码块，标注行号。

### 4. 关键数据源

| 源表 | 源字段 | 中文名 | 说明 |
|------|--------|--------|------|
| SMTMODS.L_xxx | MATURITY_DT | 原始到期日期 | 监管集市第x字段 |

加上 FROM 子句的表连接关系（主表、左连表、过滤条件）。

### 5. 取数逻辑详解（分业务场景）

逐一解释每个 WHEN 分支的**业务含义**，用通俗语言说明每种场景下取什么值、为什么。

例如：
- **场景1：正常/展期/延期（非缩期）** → 取 `MATURITY_DT`（原始到期日期）
- **场景2：缩期** → 取 `MATURITY_DT_BEFORE`（延期前到期日，即缩期前的原到期日）
- **场景3：原逻辑（已废弃）** → 展示被注释掉的旧代码，说明改造原因和需求编号

### 6. 核心业务含义总结

一句话概括该字段的本质含义。

## 业务场景解释指南

金融数据字段的 CASE WHEN 逻辑往往对应以下常见业务场景：

| 场景 | 中文名 | 含义 | 常见SQL模式 |
|------|--------|------|-------------|
| 展期 | 贷款展期 | 借款人申请延长贷款期限 | `EXTENDTERM_FLG = 'Y'` 时取 `ACTUAL_MATURITY_DT` |
| 缩期 | 贷款缩期 | 借款人提前还款缩短期限 | `MATURITY_DT_BEFORE > MATURITY_DT` 时取 `MATURITY_DT_BEFORE` |
| 核销 | 贷款核销 | 贷款被核销 | `CANCEL_FLG = 'N'` 过滤 |
| 逾期 | 贷款逾期 | 未按时还款 | `OD_FLG = 'Y'` |
| 资产证券化 | 贷款证券化(ABS) | 贷款被资产证券化（打包出售） | `LOAN_STOCKEN_DATE IS NOT NULL` 过滤 |
| 结清 | 贷款结清 | 贷款正常结清，不再存续 | `ACCT_STS = '3'` 过滤排除 |
| 固定利率 | 固定利率 | 利率固定 | `INT_RATE_TYP = 'F'` |
| 浮动利率 | 浮动利率 | 利率浮动 | `INT_RATE_TYP <> 'F'` |

## 交叉引用模式

当解释字段取数逻辑时，需要交叉引用四类信息：

```
字段名（接口表字段）
  ↓ 查源码解析文件
CASE WHEN 逻辑
  ↓ 追溯源字段
监管集市表字段（L_ACCT_LOAN.MATURITY_DT）
  ↓ 查表定义文件
字段中文名 + 码表说明（"原始到期日期"）
  ↓ 结合原文
监管机构报送规范说明（"贷款借据中约定的贷款到期日期"）
```

## 双源验证模式（实现 vs 规范）

**场景**：当用户问字段取数逻辑时，最佳实践是**同场查看两个源头**：

1. **源码实现**（三层路由 → 源码解析 → CASE WHEN 逻辑）—— 实际怎么做
2. **监管原文**（参考资料/金融基础数据系统/下的原文文件）—— 人行要求怎么做

**操作步骤**：

1. 按常规三层路由找到字段的 CASE WHEN 逻辑（源码实现）
2. **同时**读取原文文件中该字段的"填报说明"列（监管要求）
3. 对比两者，逐条检查：
   - 是否覆盖了人行要求的全部业务场景（展期、缩期、正常、逾期等）
   - 是否存在实现与规范不一致的地方（如规范要求缩期必填但源码未处理）
   - 是否有额外的实现细节未在规范中明确

**输出格式示例**（在"取数逻辑详解"之后增加一段）：

```
### 与人行要求的对齐分析

| 人行规范要求 | 源码实现 | 对齐状态 |
|---|---|---|
| 展期→展期后到期日，必填 | `ZQ.EXTENDTERM_FLG='Y'` → `ACTUAL_MATURITY_DT` | ✅ 对齐 |
| 缩期→缩期后到期日，必填 | 未处理，返回 NULL | ⚠️ 偏差：实现未覆盖缩期场景 |
| 正常→不填 | 返回 NULL | ✅ 对齐 |

注：缩期场景的到期日已通过 LOAN_DUE_DATE 字段另行填报。
```

**为什么需要双源验证**：
- 用户（业务分析员/合规人员）最关心的就是"实现的跟人行要求是否一致"
- 从源码直接回答只能覆盖"怎么做"，回答不了"是否做对了"
- 双源验证能在同一轮对话中暴露对齐偏差，驱动后续修复决策

## 源码缺失时的推导技术

当实体文件引用的源码文件在 `源码/` 或 `源码解析/` 中缺失时，仍可通过以下方式推导取数逻辑：

### 1. 监管集市表字段定义 + 码表交叉验证

从实体文件中获取**业务筛选条件**（如 `ACCT_STS <> '3'`, `CANCEL_FLG <> 'Y'`），然后：

1. 查 `参考资料/监管集市/` 下对应表的字段定义文档，获取字段中文名和码表编码
2. 查码表明细（文档最后一节），理解每个码值的真实业务含义
3. 与实体文件中的描述对比，纠正可能不准确的描述

**示例**：
- 实体文件写"非注销状态"→ 查码表 A0005 发现 `3 = 结清`（非"注销"）→ 纠正为"排除已结清"
- 实体文件写"未转让"→ 查 `LOAN_STOCKEN_DATE` 字段定义发现是"证券化日期"→ 纠正为"未做资产证券化"

### 2. 同类型报表的源码类比参照

如果缺失的源码与某个已存在源码的报表业务类型相近（如同属"五篇大文章"系列），可参照其已存在的源码结构进行推断，但需标注"此为推断，不一定完全准确"。

### 3. 应用层特殊处理作为线索

即使主程序源码缺失，应用层特殊处理 SQL（如 `五大篇章-绿色贷款刷汇总金额利率.sql`）通常仍存在，可从中获取后续处理逻辑的线索。

## 历史变更记录解读

实体文件中的"历史变更记录"表格提供了重要线索：

| 属性 | 含义 |
|------|------|
| 编号 | 需求编号（可搜索对应工单） |
| 日期 | 变更实施日期 |
| 修改人 | 代码实施者 |
| 提出人 | 业务方/需求提出者 |
| 原因 | 变更类型（新建/改造/优化/修复） |

- **"新建"** 变更通常对应初始版本，源码中的 CASE WHEN 逻辑相对简单
- **"改造"** 变更通常有新旧两套逻辑交替（会留下注释掉的旧代码）
- 同一需求编号跨多个实体出现，说明是一次批量改造

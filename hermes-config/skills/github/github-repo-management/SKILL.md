# GitHub Repository Management

Create, clone, fork, configure, and manage GitHub repositories. Each section shows `gh` first, then the `git` + `curl` fallback.

## Prerequisites

- Authenticated with GitHub (see `github-auth` skill)

### Setup

```bash
if command -v gh &>/dev/null && gh auth status &>/dev/null; then
  AUTH="gh"
else
  AUTH="git"
  if [ -z "$GITHUB_TOKEN" ]; then
    if [ -f ~/.hermes/.env ] && grep -q "^GITHUB_TOKEN=" ~/.hermes/.env; then
      GITHUB_TOKEN=$(grep "^GITHUB_TOKEN=" ~/.hermes/.env | head -1 | cut -d= -f2 | tr -d '\n\r')
    elif grep -q "github.com" ~/.git-credentials 2>/dev/null; then
      GITHUB_TOKEN=$(grep "github.com" ~/.git-credentials 2>/dev/null | head -1 | sed 's|https://[^:]*:\([^@]*\)@.*|\1|')
    fi
  fi
fi

# Get your GitHub username (needed for several operations)
if [ "$AUTH" = "gh" ]; then
  GH_USER=$(gh api user --jq '.login')
else
  GH_USER=$(curl -s -H "Authorization: token $GITHUB_TOKEN" https://api.github.com/user | python3 -c "import sys,json; print(json.load(sys.stdin)['login'])")
fi
```

If you're inside a repo already:

```bash
REMOTE_URL=$(git remote get-url origin)
OWNER_REPO=$(echo "$REMOTE_URL" | sed -E 's|.*github\.com[:/]||; s|\.git$||')
OWNER=$(echo "$OWNER_REPO" | cut -d/ -f1)
REPO=$(echo "$OWNER_REPO" | cut -d/ -f2)
```

## 1. Cloning Repositories

Cloning is pure `git` — works identically either way:

```bash
# Clone via HTTPS (works with credential helper or token-embedded URL)
git clone https://github.com/owner/repo-name.git

# Clone into a specific directory
git clone https://github.com/owner/repo-name.git ./my-local-dir

# Shallow clone (faster for large repos)
git clone --depth 1 https://github.com/owner/repo-name.git

# Clone a specific branch
git clone --branch develop https://github.com/owner/repo-name.git

# Clone via SSH (if SSH is configured)
git clone git@github.com:owner/repo-name.git
```

**With gh (shorthand):**

```bash
gh repo clone owner/repo-name
gh repo clone owner/repo-name -- --depth 1
```

## 2. Creating Repositories

**With gh:**

```bash
# Create a public repo and clone it
gh repo create my-new-project --public --clone

# Private, with description and license
gh repo create my-new-project --private --description "A useful tool" --license MIT --clone

# Under an organization
gh repo create my-org/my-new-project --public --clone

# From existing local directory
cd /path/to/existing/project
gh repo create my-project --source . --public --push
```

**With git + curl:**

```bash
# Create the remote repo via API
curl -s -X POST \
  -H "Authorization: token $GITHUB_TOKEN" \
  https://api.github.com/user/repos \
  -d '{
    "name": "my-new-project",
    "description": "A useful tool",
    "private": false,
    "auto_init": true,
    "license_template": "mit"
  }'

# Clone it
git clone https://github.com/$GH_USER/my-new-project.git
cd my-new-project

# -- OR -- push an existing local directory to the new repo
cd /path/to/existing/project
git init
git add .
git commit -m "Initial commit"
git remote add origin https://github.com/$GH_USER/my-new-project.git
git push -u origin main
```

To create under an organization:

```bash
curl -s -X POST \
  -H "Authorization: token $GITHUB_TOKEN" \
  https://api.github.com/orgs/my-org/repos \
  -d '{"name": "my-new-project", "private": false}'
```

### Initializing a Local Repo from an Existing Project (no GitHub yet)

When the user has an existing project (like a repo of SQL/DDL scripts, documents, or code) with no version control at all. This is not about creating a GitHub repo — it's about establishing a local Git baseline first.

#### Step 0: Verify it's not already a repo

```bash
git status 2>&1
# → "fatal: not a git repository" → proceed
# → anything else → already tracked, don't init again
```

#### Step 1: Create a project-tailored .gitignore

Analyze the project's file types before writing. Common patterns for SQL/documentation projects:

```gitignore
# System files
Thumbs.db
.DS_Store
Desktop.ini

# IDE/editor
.vscode/
.idea/
*.swp
*.swo

# Temp files
*.tmp
*.log
*.bak

# Build/cache
__pycache__/
*.py[cod]
```

**Always show the .gitignore content to the user as a preview** before writing it. Say something like: *"Here's what I'm planning to exclude from version control — confirm before I proceed."* (This respects user-profile requirements for preview-before-execute.)

#### Step 2: Initialize the repo

```bash
cd /path/to/project
git init              # Creates .git/ in the project root
git branch -m main    # Rename default branch from 'master' to 'main'
```

💡 **Pitfall**: `git init` creates a branch called `master` by default. Rename it to `main` immediately unless the user explicitly prefers something else.

#### Step 3: Handle author identity

The first commit will fail with `Author identity unknown` if no `user.email`/`user.name` is configured. **Ask the user** for their name and email first. If they're unsure or want temporary info:

```bash
# Local config only (not --global) — affects just this repo
cd /path/to/project
git config user.email "username@localhost"
git config user.name "username"
```

Pitfall: Setting `--global` will affect all repos. Default to repo-local config and tell the user they can update it later with `git config user.name "Real Name"`.

#### Step 4: Stage and commit

```bash
git add .                          # Stage all files (minus .gitignore exclusions)
git commit -m "初始化：项目名称与描述"
```

**Pitfall with Chinese filenames on WSL/NTFS**: `git add .` may show weird escape sequences in output (e.g. `\345\256\236\344\275\223`) but this is cosmetic only — the files are tracked correctly. Don't try to flag this as an error.

#### Step 5: Show the result

```bash
git log --oneline --graph
# → should show one commit with a SHA
```

#### Pitfalls (capture these proactively)

| Pitfall | Symptom | Fix |
|---------|---------|-----|
| Default branch is `master` | `git init` prints hint about default branch | `git branch -m main` |
| Missing author identity | `fatal: empty ident name` | Set `user.email` + `user.name` (local config) |
| User hasn't specified name | Author identity unknown | Ask before committing; don't guess |
| .gitignore not created first | Vendor dirs, IDE files, OS files get committed | Always write .gitignore before `git add .` |
| Files too large for Git | Warning about file size | Add to .gitignore, or tell user `git lfs` exists |
| Non-ASCII repository name rejected | `fatal: remote error: is not a valid repository name` | GitHub only allows ASCII alphanumeric, hyphens (`-`), underscores (`_`), and dots (`.`) in repo names. Chinese, Cyrillic, emoji, etc. are rejected. Translate to English (e.g. `financial-data-reporting`) or use pinyin abbreviation (e.g. `jrsjbs`) |

See `references/gitignore-patterns.md` for project-type-specific .gitignore examples.

## 3. Forking Repositories

**With gh:**

```bash
gh repo fork owner/repo-name --clone
```

**With git + curl:**

```bash
# Create the fork via API
curl -s -X POST \
  -H "Authorization: token $GITHUB_TOKEN" \
  https://api.github.com/repos/owner/repo-name/forks

# Wait a moment for GitHub to create it, then clone
sleep 3
git clone https://github.com/$GH_USER/repo-name.git
cd repo-name

# Add the original repo as "upstream" remote
git remote add upstream https://github.com/owner/repo-name.git
```

### Keeping a Fork in Sync

```bash
# Pure git — works everywhere
git fetch upstream
git checkout main
git merge upstream/main
git push origin main
```

**With gh (shortcut):**

```bash
gh repo sync $GH_USER/repo-name
```

  "https://api.github.com/search/repositories?q=machine+learning+language:python&sort=stars&per_page=10" \
  | python3 -c "

for r in json.load(sys.stdin)['items']:

## 5. Repository Settings

**With gh:**

```bash
gh repo edit --description "Updated description" --visibility public
gh repo edit --enable-wiki=false --enable-issues=true
gh repo edit --default-branch main
gh repo edit --add-topic "machine-learning,python"
gh repo edit --enable-auto-merge
```

**With curl:**

```bash
curl -s -X PATCH \
  -H "Authorization: token $GITHUB_TOKEN" \
  https://api.github.com/repos/$OWNER/$REPO \
  -d '{
    "description": "Updated description",
    "has_wiki": false,
    "has_issues": true,
    "allow_auto_merge": true
  }'

# Update topics
curl -s -X PUT \
  -H "Authorization: token $GITHUB_TOKEN" \
  -H "Accept: application/vnd.github.mercy-preview+json" \
  https://api.github.com/repos/$OWNER/$REPO/topics \
  -d '{"names": ["machine-learning", "python", "automation"]}'
```

## 7. Secrets Management (GitHub Actions)

**With gh:**

```bash
gh secret set API_KEY --body "your-secret-value"
gh secret set SSH_KEY < ~/.ssh/id_rsa
gh secret list
gh secret delete API_KEY
```

**With curl:**

Secrets require encryption with the repo's public key — more involved via API:

```bash
# Get the repo's public key for encrypting secrets
curl -s \
  -H "Authorization: token $GITHUB_TOKEN" \
  https://api.github.com/repos/$OWNER/$REPO/actions/secrets/public-key

# Encrypt and set (requires Python with PyNaCl)
python3 -c "
from base64 import b64encode
from nacl import encoding, public
import json, sys

# Get the public key
key_id = '<key_id_from_above>'
public_key = '<base64_key_from_above>'

# Encrypt
sealed = public.SealedBox(
    public.PublicKey(public_key.encode('utf-8'), encoding.Base64Encoder)
).encrypt('your-secret-value'.encode('utf-8'))
print(json.dumps({
    'encrypted_value': b64encode(sealed).decode('utf-8'),
    'key_id': key_id
}))"

# Then PUT the encrypted secret
curl -s -X PUT \
  -H "Authorization: token $GITHUB_TOKEN" \
  https://api.github.com/repos/$OWNER/$REPO/actions/secrets/API_KEY \
  -d '<output from python script above>'

# List secrets (names only, values hidden)
curl -s \
  -H "Authorization: token $GITHUB_TOKEN" \
  https://api.github.com/repos/$OWNER/$REPO/actions/secrets \
  | python3 -c "
import sys, json
for s in json.load(sys.stdin)['secrets']:
    print(f\"  {s['name']:30}  updated: {s['updated_at']}\")"
```

Note: For secrets, `gh secret set` is dramatically simpler. If setting secrets is needed and `gh` isn't available, recommend installing it for just that operation.

    "tag_name": "v1.0.0",

## 9. GitHub Actions Workflows

**With gh:**

```bash
gh workflow list
gh run list --limit 10
gh run view <RUN_ID>
gh run view <RUN_ID> --log-failed
gh run rerun <RUN_ID>
gh run rerun <RUN_ID> --failed
gh workflow run ci.yml --ref main
gh workflow run deploy.yml -f environment=staging
```

**With curl:**

```bash
# List workflows
curl -s \
  -H "Authorization: token $GITHUB_TOKEN" \
  https://api.github.com/repos/$OWNER/$REPO/actions/workflows \
  | python3 -c "
import sys, json
for w in json.load(sys.stdin)['workflows']:
    print(f\"  {w['id']:10}  {w['name']:30}  {w['state']}\")"

# List recent runs
curl -s \
  -H "Authorization: token $GITHUB_TOKEN" \
  "https://api.github.com/repos/$OWNER/$REPO/actions/runs?per_page=10" \
  | python3 -c "
import sys, json
for r in json.load(sys.stdin)['workflow_runs']:
    print(f\"  Run {r['id']}  {r['name']:30}  {r['conclusion'] or r['status']}\")"

# Download failed run logs
RUN_ID=<run_id>
curl -s -L \
  -H "Authorization: token $GITHUB_TOKEN" \
  https://api.github.com/repos/$OWNER/$REPO/actions/runs/$RUN_ID/logs \
  -o /tmp/ci-logs.zip
cd /tmp && unzip -o ci-logs.zip -d ci-logs

# Re-run a failed workflow
curl -s -X POST \
  -H "Authorization: token $GITHUB_TOKEN" \
  https://api.github.com/repos/$OWNER/$REPO/actions/runs/$RUN_ID/rerun

# Re-run only failed jobs
curl -s -X POST \
  -H "Authorization: token $GITHUB_TOKEN" \
  https://api.github.com/repos/$OWNER/$REPO/actions/runs/$RUN_ID/rerun-failed-jobs

# Trigger a workflow manually (workflow_dispatch)
WORKFLOW_ID=<workflow_id_or_filename>
curl -s -X POST \
  -H "Authorization: token $GITHUB_TOKEN" \
  https://api.github.com/repos/$OWNER/$REPO/actions/workflows/$WORKFLOW_ID/dispatches \
  -d '{"ref": "main", "inputs": {"environment": "staging"}}'
```

## 11. Auto-Push on Commit (Post-Commit Hook)

Configure a local repo to automatically push to GitHub every time you commit. This eliminates the need to manually run `git push` after each commit — useful for single-developer workflows where every commit should go straight to the remote.

### Prerequisites

- Remote `origin` must be configured (`git remote -v` shows it)
- Authentication must work without prompts (SSH key or credential helper — see `github-auth` skill)
- Current branch must have an upstream tracking branch (`git branch -vv` shows `[origin/main]`)

### Setup

```bash
# Create the post-commit hook
cat > .git/hooks/post-commit << 'EOF'
#!/bin/bash
git push origin "$(git rev-parse --abbrev-ref HEAD)" &
EOF

chmod +x .git/hooks/post-commit
```

The hook uses `git rev-parse --abbrev-ref HEAD` to push the current branch rather than hardcoding `main`, so it works on any branch. The trailing `&` backgrounds the push so the commit returns instantly — critical for repos with many files or slow networks where `git push` can take minutes.


### Verification

```bash
# Test by making a small commit
echo "test" >> /tmp/test-auto-push && git add . && git commit -m "test: verify auto-push hook"

# The push output should appear inline during the commit
# Check GitHub to confirm the commit arrived
```

### Removing the Hook

```bash
rm .git/hooks/post-commit
```

### Pitfalls

| Pitfall | Symptom | Fix |
|---------|---------|-----|
| No upstream branch set | `fatal: The current branch has no upstream branch` | `git push -u origin <branch>` first |
| Auth not configured | `Permission denied (publickey)` or password prompt | Set up SSH key or credential helper (see `github-auth`) |
| Push blocks commit on failure | Commit appears to hang or show errors | The trailing `&` in the hook backgrounds the push, so commits return instantly regardless of push speed or failure. Without `&`, a slow push (large repo, slow network, NTFS filesystem) blocks the terminal for minutes |
| GitHub normalizes username case | `git remote -v` shows `WM724796434` but push output says `remote: This repository moved. Please use the new location: git@github.com:wm724796434/...` | GitHub usernames are case-insensitive but normalized to lowercase in some contexts. Both casings work, but for consistency use lowercase in remote URLs |
| Hook created with wrong permissions | No push happens, no error | Ensure `chmod +x` was run |
| Working on a different branch | Push targets wrong branch | The `$(git rev-parse --abbrev-ref HEAD)` pattern handles this — don't hardcode branch names |

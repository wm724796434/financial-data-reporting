# 垂类 Agent 五层架构理论与本项目定位

> 本文档记录 2026-05-27 对话中关于"通用 Agent vs 垂类 Agent"、"规则引擎与文件路由"、"Ground Truth 数据集与自动回归测试"等话题的核心结论和架构模型。

## 一、垂类 Agent 五层架构模型

```
┌──────────────────────────────────────────────┐
│  5. 领域安全 & 合规层                         │  ← 审计日志、数据脱敏、回答合规校验
├──────────────────────────────────────────────┤
│  4. 领域专用工具 & 执行层                     │  ← SQL 解析器、数据血缘追踪、变更影响分析
├──────────────────────────────────────────────┤
│  3. 领域评估 & 质量保障层                     │  ← Ground truth 数据集、自动回归测试、量化指标
├──────────────────────────────────────────────┤
│  2. 领域知识 & 检索层                         │  ← 规则路由 + 实体文件 + 源码索引 + Embedding 兜底
├──────────────────────────────────────────────┤
│  1. 通用推理引擎                              │  ← LLM（不变，可以直接复用）
└──────────────────────────────────────────────┘
```

当前项目（金融基础数据报送系统）现状：
- 第 1 层：直接复用 DeepSeek-v4
- 第 2 层：部分实现（三层查询路由 / AGENTS.md 规则 / 实体文件），缺少 embedding 兜底、术语同义词映射、结构化数据血缘图
- 第 3 层：严重不足（仅有人工交叉验证和占位符扫描），缺少 Ground Truth 数据集和自动回归测试
- 第 4 层：完全缺失（本 skill 的 sqlglot MCP server 是第一个领域专用工具）
- 第 5 层：完全缺失

## 二、通用 Agent + 规则 + 知识库 ≠ 垂类 Agent

当前项目本质上是"通用 Agent（DeepSeek-v4）+ 领域规则约束（AGENTS.md/SOUL.md）+ 领域知识库（实体文件/源码解析/监管集市定义）"。这不是真正的垂类 Agent——真正垂类 Agent 需要每个层次都做领域化改造。

但当前方案有其合理性：成本极低、迭代极快，对于内部辅助查询场景效果已够用。

## 三、规则引擎与文件路由

核心思想：**把 Agent 工作流中"该读哪个文件"这一步从 LLM 的概率性判断中剥离，交给确定性系统。**

### LLM 做路由的三个本质缺陷
1. **幻觉**：LLM 可能"创造出"不存在的文件路径
2. **不一致**：同一个问题两次可能走不同的路由（模型切换后尤其严重）
3. **不可审计**：无法追溯"为什么选了文件 A 而不是文件 B"

### 规则引擎的四张映射表
```
表1：别名映射（人工维护）  "数字贷款" → "五篇大文章-3.10"
表2：程序名→实体映射（从 index.md 自动提取）
表3：表名→实体映射（从 SQL 源码自动提取）
表4：实体→文件路径映射（从 index.md 自动提取）
```

### 三级模糊匹配策略
1. 精确匹配 → 直接返回
2. 分词+交集匹配 → 返回 + "疑似"标记
3. 编辑距离匹配 → 低置信度，走降级策略（候选列表让用户选 or 全部注入 LLM）

## 四、Ground Truth 数据集

### 定义
不是训练数据，是**验收标准**（类比软件工程的单元测试）。标注的是 Agent 的"行为和事实"，而非"措辞"。

### 三级用例
1. **路由正确性**（30-40条，纯规则判定）：Agent 是否按 index.md → 实体文件 → 源码解析的顺序走
2. **取数范围完整性**（30-50条，结构化判定）：输出的 WHERE 子句是否包含所有过滤条件
3. **语义完整性**（20-30条，LLM+rubric 判定）：回答是否全面、正确、没有遗漏

### 自动回归测试
- 快速模式（每次改规则后）：只跑路由正确性，2-3 分钟
- 完整模式（每日/每周）：跑全部用例，30-60 分钟
- 核心价值：检测退步 + 变更影响分析

## 五、实时正确性保障 vs 事后回归测试

这是两个不同的问题：
- **事后**（改完规则，已有能力有没有退步？）→ 回归测试
- **事中**（用户现场提问，这次回答对不对？）→ 实时保障

### 实时保障的多层策略
1. **确定性信息检索**：规则引擎做路由，LLM 只做理解
2. **工具化事实提取**：SQL 解析器提取 WHERE 条件，LLM 不"读代码"
3. **实时结构化校验**：回答前校验表名/字段名是否存在、取数范围是否完整
4. **禁止回答策略**：高风险问题白名单，超出能力边界就明确拒绝

### 核心原则
**LLM 的本质是概率性的，监管要求是确定性的。正确的策略是把 LLM 的概率性推理局限在尽可能小的范围内，用确定性系统包围它。**

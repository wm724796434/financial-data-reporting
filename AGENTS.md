# 金融基础数据报送系统 — Agent 查询指南

本仓库为金融基础数据报送系统项目，包含 EAST/PBOCD 等监管报送系统的存储过程和 SQL 脚本。项目已纳入 Git 版本控制。当前收录 **53 个报表实体**、**69 个源码文件**、**191 张监管集市表定义**。

## 目录结构

| 目录 | 用途 |
|------|------|
| `首页/` | **一级路由入口**：index.md — 53个报表实体的完整索引 |
| `实体/` | **基于金融基础数据系统原文**的报表实体：每个实体文件按**两部分**组织——第一部分为发文原文要求（人行规范层），第二部分为代码取数业务范围（实现层） |
| `源码解析/` | 带增强注释的源码副本（加工层存储 .prc / 加工层特殊处理 .sql / 应用层特殊处理 .sql），共 69 个文件 |
| `参考资料/发文原文/` | 金融基础数据采集规范 V2.1 原文（49个）+ 实体文件引用来源 |
| `参考资料/监管集市/` | 监管集市 191 个表的字段级定义文档 |
| `参考资料/数据源审计/` | 数据源合规性审计报告 |
| `源码/` | **原始源码备份（不允许修改）** |

根目录还包含：
- `金融基础数据采集规范V2.1.pdf` — 人行原文完整版 PDF（约 2MB）
- `AGENTS.md` — 本文件（Agent 查询指南）
- `.gitignore` — Git 忽略规则

## 三层查询流程

查询任何关于表、程序、取数逻辑的问题，必须遵循以下三层路由：

### 第一层：index.md（一级路由）

**作用**：定位目标报表/程序，获取实体文件路径。

- **场景A**：用户问"XX报表是干什么的"
  → 查 `首页/index.md`，按**业务域分组**（五篇大文章 12 个 / 其他 41 个）找到目标报表
  → 点击实体文件链接进入第二层

- **场景B**：用户问"XX程序（存储过程）的功能"
  → 查 `首页/index.md` 末尾的"**按源码文件反向索引**"章节
  → 找到该程序名，看它属于哪些报表，点击对应实体文件链接

- **场景C**：用户问"某个表字段的取数逻辑"
  → 先走第一层找到报表实体 → 再走第二层

**禁止**：在未查 index.md 之前直接用 glob 搜索文件名或 grep 搜索内容。index.md 已收录全部 69 个源码文件到 53 个报表实体的映射关系，是第一查询入口。

### 第二层：实体文件（两分结构）

**作用**：获取该报表的**发文要求**和**代码实现**两个维度的信息。

- 根据 index.md 中的链接读取 `实体/<报表名_接口表名>.md`
- 实体文件按**两部分**组织：

#### 第一部分：发文原文要求（人行规范层）

> 回答"**这个表要求报送什么**"、"**新增业务应该报到哪个表**"等问题时查这里

- **官方报送范围**：人行发文定义的表级业务说明
- **纳入与排除口径**：发文明确包含和不包含的业务
- **关键业务要素**：要求报送的内容维度（客户、合同、金额、利率等）
- **字段清单**：发文定义的字段及填报规则摘要（完整信息指引到原文文件）
- 来源：`参考资料/发文原文/` 下的原文文件

#### 第二部分：代码取数业务范围（实现层）

> 回答"**这个表怎么取数**"、"**取了哪些业务**"、"**业务变更对金数有什么影响**"等问题时查这里

- **数据生成程序**：主存储过程（加工层存储 .prc）
- **监管集市表来源**：引用到的 SMTMODS 表及各自角色
- **业务筛选条件**：代码中 WHERE 条件（科目范围、客户类型范围、排除条件、产品类型映射）
- **特殊处理规则**：加工层特殊处理 + 应用层特殊处理 SQL 中的业务规则
- **历史变更记录**：需求修改历史（编号、日期、修改人、修改内容）
- **涉及源码文件**：按角色（数据生成 / 数据修正 / 数据引用）列出

**限制**：实体文件回答报表的"概要性取数逻辑"（取了哪些业务范围和筛选条件），不包含字段级精确取数逻辑（如某个字段具体从哪个源字段映射）。字段级问题需进入第三层。

### 第三层：源码解析文件（字段级取数逻辑）

**作用**：当用户需要知道某个表字段的具体取数逻辑时，深入源码查找。

- 读取 `源码解析/<子目录>/<文件名>`（.prc 或 .sql）
- 源码文件头部注释包含：程序名、业务域、用途、输出接口表、引用的监管集市表列表、修改历史
- 从 SQL 代码中查找具体的字段映射逻辑（如 `SELECT A.LOAN_NUM, A.ACCT_NUM...` 或 `WHERE A.FUND_USE_LOC_CD = 'I'`）
- 结合 `参考资料/监管集市/` 下的表定义文档获取字段的中文名和码表说明

## 查询规范

1. **分层回答**：回答概括性问题时，先用第二层实体文件的内容输出概述，只有用户追问字段细节时才进入第三层源码文件
2. **禁止深度扫描**：回答"XX报表是做什么的"这类问题，不得通读完整源码文件，应依赖 index.md 和实体文件
3. **交叉验证**：如果用户发现实体文件内容与源码不一致，以源码为准，并更新实体文件
4. **编码处理**：源码解析目录下的 .prc 文件为 UTF-8 编码，可直接读取；原始备份为 GBK 编码

## 禁止行为

- 禁止在未查 `首页/index.md` 的情况下直接用 glob/grep 搜索文件
- 禁止回答概括性问题时通读完整源码解析文件（应先用实体文件回答概述）
- 禁止回答时直接罗列函数/字段/文件路径作为第一层输出，必须先给出用户视角的描述

## 数据源审计

### 审计原则

原则上所有数据应从**监管集市（`SMTMODS` schema）** 取数。`SMTMODS` 是经过数据治理的统一数据源，直接引用上游业务系统或其他程序的输出接口表会导致数据链路不清晰、耦合度高。

### 审计方法

当需要审计数据源合规性时，执行以下步骤：

1. **扫描全部源码文件**，提取所有 FROM/JOIN/INSERT/MERGE 语句中的表引用
2. **按 schema 分类**：
   - `SMTMODS.xxx` → ✅ 合规（监管集市）
   - `CBRC_DATACORE.xxx` → ✅ 合规（银保监集市）
   - `PBOCD_DATACORE.xxx` → ⚠️ 目标/中间表，需判断是否为数据源或自身输出
   - 裸表名（无 schema 前缀） → ❓ 需逐个排查来源
3. **标记违规程序**：完全不引用 `SMTMODS` 的程序
4. **检查引用链**：使用接口表（`JS_xxx`）作为数据源的程序，应改为直接从 SMTMODS 取数

### 审计命令参考

```bash
# 查找所有带 schema 前缀的表引用
grep -rohP '\b[A-Z_]+\.\w+' --include="*.prc" --include="*.sql" . | sort -u

# 提取 FROM/JOIN 中的 SMTMODS 表
grep -rohP '(FROM|JOIN)\s+SMTMODS\.\w+' --include="*.prc" --include="*.sql" .

# 提取 FROM/JOIN 中的裸表名（无 schema 前缀）
grep -rohP '(FROM|JOIN)\s+(\w+)\s' --include="*.prc" --include="*.sql" .

# 查找完全不引用 SMTMODS 的程序
for f in $(find . -name "*.prc" -o -name "*.sql"); do
    if [ "$(grep -c 'SMTMODS\.' "$f")" -eq 0 ]; then echo "$f"; fi
done
```

### 分类标准

| 标记 | 含义 | 处理方式 |
|------|------|---------|
| ✅ 合规 | FROM/JOIN 直接引用 `SMTMODS.表名` | 通过 |
| ⚠️ 间接引用 | 数据最终来自 SMTMODS，但经过中间表（如 `L_CUST_C_TMP`） | 建议优化为 SMTMODS 直取 |
| ❓ 存疑 | 数据源不确定是否从 SMTMODS 派生 | 需人工核查 |
| ❌ 违规 | 完全不引用 SMTMODS 或从非 SMTMODS 表直取 | 必须整改 |

### 最新审计结果

最新完整审计报告见：[参考资料/数据源审计/数据源审计报告.md](./参考资料/数据源审计/数据源审计报告.md)

- **间接引用表**：7 张表不在 SMTMODS schema 下但数据来自 SMTMODS
- **配置/映射表**：5 张（可接受）

# Entity File Organization — 实体文件编写规范

> **源文件**: 本 skill 的源文件在项目仓库 `.hermes/skills/entity-file-organization/SKILL.md`，随版本控制。

---

## 触发条件

遇到以下情况时加载本技能：

- 从 0 到 1 创建新的实体文件
- 批量调整/优化现有实体文件
- 用户问"实体文件该怎么写"或"实体文件规范是什么"
- 审计后发现实体文件质量不合格，需要修复
- 需要交叉验证实体文件内容与源码是否一致

---

## 一、实体文件整体结构

每个实体文件按 **两部分** 组织，适用所有报表：

```
# <报表中文名_接口表名>

> **来源**：[<原文文件链接>]

## 业务分类
| 属性 | 值 |
|------|-----|
| 接口表 | <接口表名> |
| 中文名 | <中文名> |

---

# 第一部分：发文原文要求（人行规范层）

> 用于回答"这个表要求报送什么"、"新增业务应该报到哪个表"等问题

## 1. 官方报送范围
## 2. 纳入与排除口径
## 3. 关键业务要素
## 4. 字段清单（指引到原文文件）

---

# 第二部分：代码取数业务范围（实现层）

> 用于回答"这个表怎么取数"、"取了哪些业务"、"业务变更对金数有什么影响"等问题

## 1. 数据生成程序
## 2. 监管集市表来源
## 3. 输出接口表
## 4. 取数筛选条件          ← 核心章节，见第三节规范
## 5. 特殊处理规则
## 6. 历史变更记录
## 7. 涉及源码文件
```

**注意**：章节编号统一为 `## 1.` `## 2.` 格式，不设多层子编号（如禁止 `### 4.1` 拆分筛选条件）。

## 二、从 0 到 1 创建流程

### Step 1：收集原始资料

1. **发文原文** → `参考资料/发文原文/` 下读取对应原文文件，获取字段定义、填报规则、码表
2. **监管集市表定义** → `参考资料/监管集市/` 下读取对应的表定义文档，获取字段类型、码表、说明
3. **源码文件** → `源码解析/加工层存储/` 下读取对应的 .prc 文件，获取实际 SQL 逻辑
4. **应用层特殊处理** → 如果存在对应的 .sql 特殊处理，一并读取

### Step 2：编写第一部分（发文原文要求）

| 章节 | 内容要求 |
|------|---------|
| **官方报送范围** | 用人行发文原文的表述，说明这个表是报送什么业务的。一句话概括。 |
| **纳入与排除口径** | 表格形式，✅ 纳入什么业务、❌ 排除什么业务。从发文原文提炼。 |
| **关键业务要素** | 报送的业务维度（如"客户维度、合同维度、金额维度、利率维度"等） |
| **字段清单** | 给出总字段数量 + 指引到原文文件。不在此处展开全部字段。 |

### Step 3：编写第二部分（代码取数实现）

按以下顺序逐一编写：

#### 第 1 节：数据生成程序

```
`<程序名>.prc`（加工层存储过程）- <一句话描述功能>
```

#### 第 2 节：监管集市表来源

表格格式，列出所有引用的 SMTMODS 表：

| 表名 | 中文名 | 角色 |
|------|--------|------|
| `SMTMODS.XXX` | xxx表 | **主表**—用途说明 |
| `SMTMODS.YYY` | yyy表 | **关联表**—关联条件说明 |

角色标识用 **加粗**，如 **主表**、**客户信息**、**汇率折算** 等。

#### 第 3 节：输出接口表

```
`PBOCD_JS_XXX` — <输出表中文名>
```

#### 第 4 节：取数筛选条件 ★（核心）

**这是最容易被写错的章节，必须严格遵循以下规则：**

##### 规则 1：合并为完整 WHERE 子句

所有 AND 条件必须放在**同一个 SQL 代码块**中输出。禁止分节写（如禁止"4.1 标签识别"+"4.2 存量过滤"+"4.3 时间范围"）。

```sql
-- ✅ 正确：所有条件合并
WHERE T.GREEN_CREDIT_FLAG = '1'       -- 绿色标志=1
  AND T.LOAN_ACCT_BAL > 0             -- 余额>0
  AND T.ACCT_STS <> '3'               -- 非结清
  AND TO_CHAR(T.DRAWDOWN_DT, 'YYYYMM') = ...  -- 时间范围
```

```sql
-- ❌ 错误：分节写
-- 4.1 绿色贷款识别（WHERE ...）
-- 4.2 存量过滤（WHERE ...）
```

##### 规则 2：区分筛选与非筛选

- **筛选条件**：参与 WHERE 子句的字段 → 写入取数筛选条件
- **非筛选字段**：仅出现在 SELECT 中透传或 CASE WHEN 映射的字段 → 写入"特殊处理规则"或另起"字段映射说明"小节，**不要与筛选条件混在一起**

> 例子：GREEN_CREDIT_USAGE 和 GREEN_CREDIT_RATIO 虽然与绿色贷款相关，但如果不参与 WHERE 过滤，就不属于"取数筛选条件"

##### 规则 3：写清码表含义

字段的码表值不能只写"非注销状态"这种模糊描述，要写清楚实际值和含义：

```sql
-- ✅ 正确
AND T.ACCT_STS <> '3'   -- 排除结清（码表：1=正常, 2=逾期, 3=结清, 9=其它）
```

```
-- ❌ 错误
AND T.ACCT_STS <> '3'   -- 非注销状态（'3'实际是结清，不是注销）
```

##### 规则 4：字段值以源码为准

实体文件中的条件值必须与源码交叉验证：

| 说法 | 实际源码 | 正确写法 |
|------|---------|---------|
| "绿色贷款标识不为空" | `GREEN_CREDIT_FLAG = '1'` | `GREEN_CREDIT_FLAG = '1'` |
| "非注销状态" | `ACCT_STS <> '3'`（3=结清） | `ACCT_STS <> '3'`（排除结清） |

**一律从源码中提取实际的 WHERE 条件，不要靠猜测或转述。**

##### 规则 5：写完整注释

每个条件后面加行内注释，说明业务含义。条件较多时，在 WHERE 子句下方追加说明段落，解释特殊情况。

#### 第 5 节：特殊处理规则

记录非 WHERE 条件类的业务规则：
- 应用层 SQL 中的汇总/刷数逻辑
- 字段映射规则（CASE WHEN / DECODE）
- 数据修正逻辑
- 历史版本变更的注释

如果是字段映射规则，用表格或代码块清晰展示映射关系。

#### 第 6 节：历史变更记录

表格格式：

| 编号 | 日期 | 修改人 | 提出人 | 原因 |
|------|------|--------|--------|------|
| JLBA... | YYYY-MM-DD | 姓名 | 姓名 | 新建/修改原因 |

#### 第 7 节：涉及源码文件

按角色列出：

| 角色 | 文件路径 |
|------|---------|
| 数据生成（主程序） | `源码/加工层存储/xxx.prc` |
| 数据生成（历史版本） | `源码/加工层存储/xxx_his.prc` |
| 应用层特殊处理 | `源码解析/应用层特殊处理/xxx.sql` |

## 三、质量检查清单

编写或修改完成后，逐项检查：

### 内容完整性

- [ ] 有来源链接指向发文原文
- [ ] 第一部分有官方报送范围
- [ ] 第一部分有纳入/排除口径（表格）
- [ ] 第二部分有数据生成程序（写明程序名和用途）
- [ ] 第二部分有监管集市表来源（表格，含角色说明）
- [ ] 第二部分有取数筛选条件（完整 WHERE 子句）
- [ ] 第二部分有特殊处理规则
- [ ] 第二部分有历史变更记录
- [ ] 第二部分有涉及源码文件

### 取数条件正确性

- [ ] WHERE 子句中所有条件合并为一条 SQL，无分散章节
- [ ] 每个条件都有行内注释说明业务含义
- [ ] 码表值写明了具体含义（不是模糊描述）
- [ ] 已与源码交叉验证，条件值和字段名与源码一致
- [ ] 筛选条件与非筛选字段已明确区分
- [ ] 不参与筛选的字段在说明中标注了"不参与筛选"

### 第二部分质量

- [ ] 监管集市表的角色说明用加粗标识，且有实质性描述（如"**主表**—贷款借据信息及绿色贷款标识"）
- [ ] 特殊处理规则包含实质性业务规则（SQL 片段或映射表）
- [ ] 历史变更记录包含至少一条记录，或者注明"暂无变更记录"

## 四、常见陷阱

| 陷阱 | 后果 | 避免方法 |
|------|------|---------|
| 按源码文件章节分段写筛选条件 | 用户看到的取数范围是割裂的 | 强制合并为一条 WHERE |
| 把 SELECT 中的字段也算作筛选条件 | 虚增条件数量，误导用户 | 只看 WHERE 子句 |
| 码表值凭印象写 | 描述错误（如"注销"写成"结清"） | 从监管集市表定义中查码表 |
| 条件值不验证直接复述 | 与源码不一致 | 读取源码解析文件交叉验证 |
| 筛选条件写在"特殊处理"中 | 用户找不到条件 | 所有 WHERE 条件归入"取数筛选条件" |

## 五、批量调整流程

当需要对多个实体文件进行统一调整时（如修复筛选条件章节），按以下流程执行：

1. **摸底**：先用 search_files 统计需要调整的文件数量，确认范围
2. **参考**：以已修正的绿色贷款文件（3.3和3.4）为参考模板
3. **分批处理**：
   - 优先处理五篇大文章（12个）— 结构相似，问题集中
   - 其次处理其他有"业务筛选条件"章节的文件
   - 最后处理无筛选条件的文件（检查是否遗漏）
4. **交叉验证**：每批处理完毕后，抽查源码确认条件值正确
5. **提交确认**：每批调整后展示变更内容，待用户确认后再继续

# 金融基础数据报送系统

本仓库为金融基础数据报送系统项目，包含 EAST/PBOCD 等监管报送系统的存储过程和 SQL 脚本。当前收录 **58 个报表实体**、**68 个源码文件**（56个 .prc + 12个 .sql）、**191 张监管集市表定义**。反向索引覆盖全部 68 个源码文件。

## 目录结构

| 目录 | 用途 |
|------|------|
| `首页/` | 一级路由入口：index.md — 58个报表实体的完整索引 + 业务路由表.md |
| `实体/` | 报表实体文件，按三部分组织：第一部分为发文原文要求（人行规范层），第二部分为代码取数业务范围（实现层），第三部分为报送指引（若有） |
| `源码解析/` | 带增强注释的源码副本（加工层存储 .prc / 加工层特殊处理 .sql / 应用层特殊处理 .sql），共 68 个文件 |
| `参考资料/发文原文/` | 金融基础数据采集规范 V2.1 原文（54个）+ 实体文件引用来源 |
| `参考资料/监管集市/` | 监管集市 191 个表的字段级定义文档 |
| `参考资料/报送指引/` | 人行报送指引（填报操作指南，目前仅单位贷款） |
| `审计/` | 数据源合规审计 + 文档质量审计报告 |
| `源码/` | 原始源码备份（不允许修改） |

## 三层查询流程

查询任何关于表、程序、取数逻辑的问题，必须遵循以下三层路由：

### 第一层：index.md（一级路由）

- **场景A**：用户问"XX报表是干什么的" → 查 `首页/index.md`，按业务域分组找到目标报表 → 点击实体文件链接进入第二层
- **场景B**：用户问"XX程序（存储过程）的功能" → 查 `首页/index.md` 末尾的"按源码文件反向索引"章节 → 找到该程序名，看它属于哪些报表
- **场景C**：用户问"某个表字段的取数逻辑" → 先走第一层找到报表实体 → 再走第二层
- **场景D**：用户问"新增XX业务应该报到哪个表" → 查 `首页/业务路由表.md`，按业务大类（存款/贷款/票据/投资/五篇大文章等）找到对应业务场景 → 点击实体文件链接

**禁止**在未查 index.md 或业务路由表之前直接用 glob 搜索文件名或 grep 搜索内容。

### 第二层：实体文件（两分结构）

根据 index.md 中的链接读取 `实体/<报表名_接口表名>.md`，实体文件按两部分组织：

- **第一部分 — 发文原文要求（人行规范层）**：回答"这个表要求报送什么"、"新增业务应该报到哪个表"等问题。包含官方报送范围、纳入与排除口径、关键业务要素、字段清单。
- **第二部分 — 代码取数业务范围（实现层）**：回答"这个表怎么取数"、"取了哪些业务"、"业务变更对金数有什么影响"等问题。包含数据生成程序、监管集市表来源、业务筛选条件、特殊处理规则、历史变更记录、涉及源码文件。

限制：实体文件回答报表的"概要性取数逻辑"，不包含字段级精确取数逻辑。字段级问题需进入第三层。

### 第三层：源码解析文件（字段级取数逻辑）

当用户需要知道某个表字段的具体取数逻辑时，深入源码查找：
- 读取 `源码解析/<子目录>/<文件名>`（.prc 或 .sql）
- 源码文件头部注释包含：程序名、业务域、用途、输出接口表、引用的监管集市表列表、修改历史
- 从 SQL 代码中查找具体的字段映射逻辑
- 结合 `参考资料/监管集市/` 下的表定义文档获取字段的中文名和码表说明

## 查询规范

1. **分层回答**：回答概括性问题时，先用第二层实体文件的内容输出概述，只有用户追问字段细节时才进入第三层
2. **禁止深度扫描**：回答"XX报表是做什么的"这类问题，不得通读完整源码文件，应依赖 index.md 和实体文件
3. **交叉验证**：如果用户发现实体文件内容与源码不一致，以源码为准，并更新实体文件
4. **编码处理**：源码解析目录下的 .prc 文件为 UTF-8 编码，可直接读取；原始备份为 GBK 编码
5. **取数范围回答规范**：回答"取数范围"类问题时，必须将所有筛选条件整合为完整的 SQL 取数范围输出（FROM + JOIN + WHERE），标注每个条件是【过滤】还是【仅字段补充】。禁止按实体文件章节分段叙述。回答后必须附上完整的 SQL 片段方便验证。需注意区分：只有参与过滤的条件才算"取数范围"，SELECT 中的字段透传和 CASE WHEN 分类映射不属于取数范围。

## 禁止行为

- 禁止在未查 `首页/index.md` 的情况下直接用 glob/grep 搜索文件
- 禁止回答概括性问题时通读完整源码解析文件（应先用实体文件回答概述）
- 禁止回答时直接罗列函数/字段/文件路径作为第一层输出，必须先给出用户视角的描述

## 持续优化机制

每轮对话结束时，必须审视本轮交互涉及的内容，检查是否存在可优化项，若有则写入 `待优化事项/待优化清单.md`。

### 审视范围

| 类别 | 检查要点 |
|------|---------|
| **实体文件问题** | 内容与源码不一致、取数范围描述错误、字段映射遗漏、缺少实质性业务规则 |
| **代码问题** | SQL 逻辑缺陷、别名引用错误、性能隐患、不规范写法 |
| **数据源合规问题** | 引用了非 SMTMODS 表但未记录、新发现的数据源违规 |
| **文档缺失** | 某报表缺少实体文件、实体文件第二部分缺少实质性业务规则 |
| **基础设施问题** | 目录结构不合理、文件编码问题、命名不规范 |
| **其他** | 任何能让项目更准确、更可维护的改进点 |

### 记录规则

1. **有则写入**：发现优化项 → 按 `待优化事项/待优化清单.md` 的表格格式填写完整
2. **无则跳过**：未发现优化项 → 无需操作
3. **禁止重复记录**：本轮对话中已经完成了的优化工作，不得再作为待优化项写入
4. **优先级判断**：影响数据报送正确性的为 P0，影响运行效率或可维护性的为 P1，可延后处理的推荐优化为 P2，纯风格问题为 P3

## 文件同步规则

**更新 CLAUDE.md 的同时必须同步更新 AGENTS.md**，确保两个文件内容保持一致。两个文件互为镜像，任何对其中一个的修改都必须反映到另一个。

AGENTS.md 位于项目根目录 `AGENTS.md`，CLAUDE.md 位于项目根目录 `CLAUDE.md`。
